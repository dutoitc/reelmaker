# Project manifest — Reelmaker

## Snapshot

- Version: **0.2.3**
- Status: local CLI MVP, suitable for a public GitHub repository as an experimental project.
- Primary platform: Windows 11 + Git Bash.
- Processing policy: local/offline by default.
- Main hardware target: NVIDIA RTX 4070 with 12 GB VRAM.
- License: MIT.

## Product objective

Generate several usable 9:16 reels from a long-form CAPStv/Xplore Swiss TV video, with human validation before final rendering.

Current input:

- YouTube URL or existing SRT/VTT for transcript timing;
- local source video for rendering;
- Ollama for candidate selection.

Current output:

- vertical H.264 MP4 reels;
- SRT and ASS subtitles;
- metadata and caption files;
- cached transcript/candidate/selection JSON;
- optional YouTube end card.

## Current pipeline

```text
YouTube/SRT transcript
  -> normalized SubtitleCue list
  -> overlapping transcript chunks
  -> Ollama candidate proposals
  -> local boundary refinement
  -> local or Ollama ranking
  -> human validation
  -> static smart crop + subtitle correction
  -> FFmpeg render
```

## Stable decisions

- Keep the CLI simple and usable from Git Bash.
- Prefer the local source video over a downloaded YouTube copy.
- Keep Ollama optional to rendering and subtitle parsing.
- Keep a human validation step before expensive rendering.
- Use JSON between expensive stages so a run can resume.
- Keep OpenCV optional; centered crop remains the fallback.
- Avoid heavy frameworks until the pipeline is stable.

## Current defaults

- Ollama model: `qwen3:4b`
- ranking: local
- candidate chunks: 300 seconds with 20 seconds overlap
- target reel duration: about 22 seconds
- output: 1080x1920, H.264/AAC
- crop: static `smart` mode, faces then motion then center
- subtitle correction: conservative `basic`

## Code health audit — 2026-06-30

Verified:

- all automated tests pass on Python 3.13;
- project compiles successfully;
- no API key, password, token, or private key was found;
- subprocess calls use argument lists, not shell interpolation;
- generated bytecode is excluded from Git and clean archives;
- GitHub Actions runs the base test suite on Python 3.10, 3.12, and 3.13.

Accepted technical debt:

1. Cache files are not yet fully fingerprinted by source content, prompt version, model, and settings.
2. One output directory should be treated as belonging to one source project.
3. `analyzer.py` and `renderer.py` each own several responsibilities and are the main refactoring hotspots.
4. Crop is static for each reel; there is no per-shot tracking.
5. Integration tests do not yet execute real FFmpeg, yt-dlp, Ollama, or GPU transcription.

## Next validated direction

The next major capability is **direct MP4 input with local transcription**.

Preferred design:

- introduce a small `TranscriptionProvider` boundary;
- keep current SRT/YouTube providers;
- add WhisperX as an optional provider;
- preserve word-level timing in a versioned transcript JSON;
- use source-file and settings fingerprints for transcription cache;
- standardize the WhisperX development environment on Python 3.11 first, without forcing the base package to import WhisperX.

Do not add prosody, diarization, scene ranking, B-roll, and music in the same iteration. Follow `ROADMAP.md`.
